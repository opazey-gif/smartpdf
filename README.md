ปกปิดเป็นข้อยกเว้น" (พรบ. ข้อมูลข่าวสารของราชการ พ.ศ. 2540)ผู้พัฒนา (Architect): OCTO_IT & Chief Data Officer (CDO)ส่วนที่ 1: Enterprise Architecture & Business Alignment1.1 กระบวนการคิดเชิงออกแบบ (Design Thinking Approach)Empathize (เข้าใจปัญหา): พนักงานและผู้บริหารใช้เวลามากในการค้นหาเอกสารระเบียบ/คำสั่ง/คู่มือ, ไม่ทราบว่าเอกสารไหนอัปเดตล่าสุด, และกังวลเรื่องการหลุดรอดของเอกสารลับDefine (กำหนดโจทย์): สร้างศูนย์กลางข้อมูล (Single Source of Truth) ที่ค้นหาด้วยเนื้อหาได้ (AI OCR) แยกสิทธิ์การเข้าถึงชัดเจน และต้องไม่เพิ่มภาระงบประมาณ (Zero-Cost Architecture)Ideate (หาทางออก): พัฒนาระบบบน Google Workspace โดยใช้ Apps Script เป็น Backend, Google Drive เป็น Data Lake, และ Google Sheets เป็น Database ควบคู่กับระบบประมวลผล AI OCR อัตโนมัติPrototype & Test: พัฒนาและปรับปรุงอย่างต่อเนื่องจนถึงเวอร์ชัน 3.7 (รองรับ Metadata-Only และ ABAC ภาครัฐ)1.2 Business Architecture & Capabilitiesระบบนี้เข้ามาช่วยยกระดับ ขีดความสามารถทางธุรกิจ (Business Capability) ขององค์กรใน 4 แกนหลัก:Document Lifecycle Management: ความสามารถในการจัดการวงจรชีวิตเอกสาร (วันที่บังคับใช้ - วันที่หมดอายุ)Knowledge Discovery: ความสามารถในการสืบค้นความรู้ระดับข้อความ (Full-text Search ผ่าน AI OCR)Data Access Governance: ความสามารถในการควบคุมการเข้าถึงข้อมูลตามชั้นความลับ (Clearance Level)Regulatory Compliance: ความสามารถในการปฏิบัติตามกฎหมาย (พรบ.ข้อมูลข่าวสารฯ และ PDPA)1.3 As-Is vs To-Be Process🔴 As-Is (อดีต/ปัจจุบัน): * เก็บไฟล์กระจัดกระจายตาม Google Drive ของแต่ละสำนักระบบแชร์ไฟล์ใช้เพียง Link แชร์ ซึ่งเสี่ยงต่อการส่งต่อให้ผู้ไม่มีสิทธิ์ค้นหาได้เฉพาะชื่อไฟล์ ไม่สามารถค้นหาเนื้อหาข้างใน PDF ได้ไม่มีระบบติดตาม (Audit Trail) ว่าใครโหลดไฟล์ความลับไปบ้าง🟢 To-Be (อนาคตด้วยระบบใหม่):มี Web Portal กลาง (แยก Portal ระหว่างงาน KM และ งานสารบรรณ)ใช้ ABAC (Attribute-Based Access Control) กรองเอกสารก่อนแสดงผลหุ่นยนต์ AI แปลง PDF เป็นข้อความให้อัตโนมัติ รองรับการนำไปใช้กับ GenAI (NotebookLM) ในอนาคตมี Dashboard สถิติแบบ Real-time และ Audit Log ทุกการเคลื่อนไหวส่วนที่ 2: Software Requirements Specification (SRS)2.1 Functional Requirements (ความต้องการทางระบบ)FR1 - Ingestion (การนำเข้า): รองรับการอัปโหลดไฟล์ PDF (ไม่เกิน 10MB) พร้อมดึงข้อมูล Hash SHA-256 ป้องกันการอัปโหลดซ้ำFR2 - Metadata-Only: รองรับการลงทะเบียน "สมุดหน้าเหลือง" สำหรับเอกสารลับ โดยไม่ต้องอัปโหลดไฟล์ตัวจริงขึ้น Cloud (Physical Record Placeholder)FR3 - AI OCR Pipeline: ระบบมี Background Trigger ทำงานทุก 5 นาที เพื่อนำไฟล์ PDF ไปถอดรหัสข้อความผ่าน Google Docs APIFR4 - Smart Catalog: ระบบค้นหาข้อมูลกรองตาม Keyword, สำนัก, และประเภทเอกสาร พร้อมรองรับ Pagination และ Export เป็น CSVFR5 - Executive Dashboard: แสดงกราฟสถิติ 4 ด้าน (สัดส่วนสำนัก [Horizontal Bar], ชั้นความลับ [Bar], ประเภทเอกสาร [Horizontal Bar], ระดับ PDPA [Doughnut])2.2 โครงสร้างข้อมูล (Data Classification & ABAC)อ้างอิงระเบียบ พรบ. ข้อมูลข่าวสารฯ:🟢 Public (สาธารณะ): เปิดเผยเชิงรุก -> ทุกคนเข้าถึงได้🔵 Internal (ใช้ภายในราชการ): คู่มือ/ระเบียบปฏิบัติ -> พนักงานทุกคนในองค์กรเข้าถึงได้ (Cross-department knowledge sharing)🟠 Confidential (ปกปิด): ข้อมูลส่วนบุคคล/ข้อยกเว้นตามกฎหมาย -> เข้าถึงได้เฉพาะเจ้าของข้อมูล (Data Owner), ผู้ที่มีสิทธิ์ในสำนักนั้น (Supervisor), หรือผู้ที่อยู่ร่วมโปรเจกต์ (Many-to-Many Intersection)🔴 Secret (ลับ): ลับสุดยอด -> ต้องมีระดับการเข้าถึง (Clearance Level) ระดับ 4 ขึ้นไปเท่านั้นจึงจะมองเห็นส่วนที่ 3: Security & Privacy Standard (การจัดการความปลอดภัย)3.1 Privacy by Design (ความเป็นส่วนตัวตั้งแต่เริ่มคิด)Proactive Protection: ระบบมีการสแกนหาตัวเลขที่คล้ายบัตรประชาชนในไฟล์ PDF เบื้องต้น หากพบจะแจ้งเตือน (🚨 Sensitive PII Detected) และปรับชั้นความลับเป็น Confidential อัตโนมัติData Minimization: จำกัดการเก็บข้อมูลที่จำเป็น แยกฟิลด์ PDPA Flag (Safe, General PII, Sensitive PII) อย่างชัดเจนOwner Accountability: การอัปโหลดเอกสารระบุ Data Owner ชัดเจน และ Owner เท่านั้น (หรือ Supervisor) ที่มีสิทธิ์แก้ไข (Edit) หรือจัดเก็บ (Archive) โดยมี Data Steward เป็นผู้ดูแลการนำเข้า3.2 Security Framework (สอดคล้องกับ OWASP Top 10)SSO & OAuth 2.0: ระบบไม่ได้สร้างกลไก Login เอง แต่ใช้ Identity Provider ของ Google Workspace ซึ่งรองรับ MFA (Multi-Factor Authentication) ระดับองค์กรA01: Broken Access Control: ระบบกรองสิทธิ์ตั้งแต่ฝั่ง Backend (Server-side) ผ่านฟังก์ชัน evaluatePolicy หากไม่มีสิทธิ์ จะไม่ส่งข้อมูลออกมาที่หน้าเว็บ (Zero-Trust Approach)A03: Injection: ใช้ Google Apps Script API ในการเขียนข้อมูลลง Sheets โดยตรง หลีกเลี่ยงการต่อสตริงที่เสี่ยงต่อ SQL/NoSQL InjectionA09: Security Logging and Monitoring: มีกลไก logAuditEvent บันทึกทุกกิจกรรม (Upload, Edit, View, Download, Archive) พร้อม Timestamp และอีเมลผู้กระทำ ลงใน Table_AuditLogs ที่แก้ไขไม่ได้ (Immutable)Quota Protection (Safe Limit): จำกัดจำนวนการประมวลผล AI ไว้ที่ MAX_PROCESS_PER_RUN = 3 เพื่อป้องกันทรัพยากรทำงานหนักเกินไปจนโดน Google ระงับโควตา (Denial of Wallet Protection)ส่วนที่ 4: Implementation & History (แผนงานและประวัติการพัฒนา)4.1 Phase Plan (แผนดำเนินงาน)Phase 1: Master Code Development - พัฒนา Template หลักให้รองรับ ABAC, OCR, และ Security Framework (เสร็จสิ้น)Phase 2: Project Separation - นำ Master Code โคลนออกเป็น 2 ระบบ (Knowledge Portal และ ระบบสารบรรณ) เพื่อแยกลอจิกและป้องกัน Context Mixing ใน AI (กำลังดำเนินการ)Phase 3: Rollout & UAT - จัดทำคู่มือ (SOP) ทดสอบระบบแบบสวมบทบาท (Roleplay Testing) และเปิดใช้งานจริงPhase 4: AI Integration - นำโฟลเดอร์ AI_Extracted_Text ไปเชื่อมต่อกับ NotebookLM เพื่อทำ AI Search Assistant ให้กับองค์กร4.2 History of Development (ประวัติการพัฒนา Master Code)v1.0 - 2.0: วางโครงสร้างพื้นฐาน Upload และ Search UI
